## 1. relay a2 trust-boundary phase 0-3

> **背景**：本期最大架构主轴。Hermes Gateway 历史上每个 messaging 平台都是一段 in-tree adapter（Telegram/Discord/Slack/Mattermost/Photon/iMessage 等），平台 webhook/crypto/identity 验证全部在 gateway 内。这一波引入 **Relay/A2** 信任边界模型：把平台 secret + crypto 边界外推到一个独立 **Connector** 进程，gateway 通过 **WebSocketRelayTransport** 与 connector 协商 **CapabilityDescriptor**，gateway 视图里只剩 tenant-scoped MessageEvent / send-action。这让多租户共享 bot identity + 临时性 hosted gateway（无 secret 持久化）成为可能。
>
> 体量：23 commit 上线 Phase 0-3 全骨架；跨 `gateway/relay/` 6 个新模块 + `docs/relay-connector-contract.md` v1 + 6 套测试 + 配置驱动激活 + Nous Portal 集成 + CLI `hermes gateway enroll` 子命令。

## 5. openviking session race wave

> **背景**：OpenViking memory provider 此前的 session 切换处理在并发 sync_turn / on_memory_write / on_session_end / on_session_switch 之间存在多个 race window：陈旧 prefetch 残留、跨 session_id 漏 write、双 commit、命令线程被同步 commit 阻塞 ~秒级（`/new` / `/branch` / `/resume` 卡顿）。本 wave **12 commit** 把这些 race 全部按生产级并发模型收口。

### 5.1 并发模型四大原语（HEAD `plugins/memory/openviking/__init__.py`）

1. **Session state lock**（`threading.Lock`）—— 保护 session 旋转：sync_turn 要么在旋转之前完整落，要么完整落到新 session，绝不跨边界
2. **Per-session writer registry** —— `_inflight_writers: Dict[str, Set[threading.Thread]]`（`:1664`）；每个 writer（sync_turn / on_memory_write）启动时 `_spawn_writer(sid, fn, ctx)`（`:2080-2102`）自注册，退出时 `finally` 自反注册（`:2093-2097`）；commit 路径调 `_drain_writers(sid, timeout)`（`:2129-2156`）等给定 sid 的**所有** writer，全部 drain 才允许 commit（`:2254-2260`）
3. **Prefetch generation counter** —— 单调 int；worker 在 spawn 时 capture 该值，长操作完成时如已增（session 已切换）→ 拒绝写
4. **Async finalization offload** —— `_finalize_session_async(...)` 在 daemon 线程跑 drain + 待 tag GET + commit POST，让命令线程立返；`/new` / `/branch` / `/resume` / `/undo` 不再卡

### 5.2 12 commit 逐条（按 wave 演化序）

`813a4e383 (#28296) fix(openviking): implement on_session_switch hook` —— wave 起点。agent 旋转 `session_id`（`/new` / `/branch` / `/reset` / `/resume` / 上下文压缩）时 provider cached `_session_id` 卡在初始化值；所有后续 sync_turn write 落到已闭 session；on_session_end 双 commit

## 9. langfuse turn scoped trace

> **背景**：同一 task/session 内并发 API 请求在 trace state 层冲突。Per-turn / per-request scoping + LRU 收尾。

### 9.1 算法

**根因**：`_trace_key()` 之前返 per-session 常量；同 gateway session 内两次 API 请求（同 `task_id == session_id`）映射同 trace key，state 合并；turn 2 复用 turn 1 残留 state → 只开一个 trace 而不是两个

**Scoping 分层**（`plugins/observability/langfuse/__init__.py:247 _trace_key`）：
```python
_trace_key(task_id, session_id, turn_id="", api_request_id="")
```
- **Tier 1（turn-scoped）**：`turn_id` 在 → `task:T:turn:U` | `session:S:turn:U` | `thread:PID:turn:U`
- **Tier 2（request-scoped）**：`api_request_id` 在 → `task:T:api:R` | `session:S:api:R` | `thread:PID:api:R`
- **Tier 3（legacy）**：fallback 到裸 `task_id` 或 `session:S` 保 backward compat
- **Turn 优先**：turn_id + api_request_id 都在时 turn_id 赢（turn-scoped `post_llm_call` 无 api_request_id，必须 resolve 到与 request hook 同 key）

**共享 prefix 抽取**（`e1d10ec1e _scope_prefix` `:231`）：
- 去重 task/session/thread fallback ladder；后续加 scope 维度不会在各 ladder 实例间漂移

**LRU 驱逐 for 非-finalizing turn**（`f4fbaa6cd` `:713`）：
- 问题：per-turn key + `_finish_trace` 只 pop 干净 turn；非-finalizing turn（被打断 / tool-only final / 空 final content）永留 entry 让 `_TRACE_STATE` 无界增长
- 修：`_evict_stale_locked()` 在 `_STATE_LOCK` 下 insert 前调（`on_pre_llm_call` / `on_pre_llm_request`），`len(_TRACE_STATE) > _MAX_TRACE_STATE` 时 LRU 驱逐最久未更新；驱逐时先结 root span 再删
- `_MAX_TRACE_STATE = 256`（远高于任何真实并发活集；仅作泄漏上限）
