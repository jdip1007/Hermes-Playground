## v0.12.0 (2026-04-30) — Curator 自治化

### Curator 升级为后台 Agent

`agent/curator.py` 在 v0.11.0 已落地，v0.12.0 把它升级为**真正的后台 Agent**：

- **运行宿主**：网关 cron-ticker 线程，默认 7 天周期；CLI/Gateway 启动也按 `interval_hours` + `min_idle_hours` 触发
- **统一配置**：`auxiliary.curator` —— 跟 vision/compression 一样的 task slot，可在 `hermes model` 里选 curator 用哪个模型，也可从 dashboard 配置
- **Per-run 报告**：`logs/curator/run.json` + `REPORT.md`
- **归档分类**：consolidated（被合并的）vs pruned（被删除的），由 model + heuristic 联合判定
- **`hermes curator status`** 现在按使用量排序，给出最常用/最不用的列表

CLI 子命令矩阵（`hermes_cli/curator.py`）：

```
hermes curator status        # 状态+技能使用排名
hermes curator run [--sync]  # 手动触发一次（v0.13: 默认同步）
hermes curator pause/resume  # 暂停/恢复
hermes curator pin/unpin     # 锁定某 skill，禁止 curator 改它
hermes curator restore       # 恢复归档技能（v0.13: 扫描嵌套子目录）
hermes curator archive       # 手动归档（v0.13 新增）
hermes curator prune         # 手动删除归档过的（v0.13 新增）
hermes curator list-archived # 列归档（v0.13 新增）
hermes curator backup/rollback  # 备份/回滚
```

**关键不变量**（在 `_pinned_guard()` 和 `skill_manage` 入口拦截）：
- 永不动 bundled/hub-installed skill
- 永不自动删除——只归档
- pinned skills 跳过所有自动转换

### 自我改进循环（Background Review）大改

- **Class-first 评分提示**：从开放式"是否需要更新"换成基于 rubric 的分级
- **active-update bias**：偏好刚刚被 agent 加载过的 skill，会处理 `references/` + `templates/` 子文件
- **Fork 继承父运行时**：provider/model/credentials 真正传递（之前会偷偷掉回默认）
- **scoped toolset**：fork 限定在 memory + skills 两个 toolset，禁止 shell/web
- **clean context**：之前一轮的 tool messages 不进 review 摘要

**生命周期**（`kanban_db.py` 工作列）：

```
todo → ready → running → blocked / done / archived
                  ↑
            heartbeat（last_heartbeat_at INTEGER）
            reclaim（dispatcher tick 检测心跳过期）
            zombie（PID 死了但 status=running）
```

**任务字段**（`Task` dataclass）：id / title / body / assignee / status / priority / tenant / workspace_kind / workspace_path / created_by / started_at / completed_at / result / skills / **max_retries**

**workspace 类型**：scratch / worktree / dir:&lt;path&gt;。多任务并发跑用 worktree 隔离防止 worker 互相覆盖。

**幻觉防护**（issue #20017 关闭）：worker 报告"我创建了卡片 X"时，dispatcher 校验该卡片是否真的写进了 db，否则触发 recovery UX。

**诊断引擎**（`kanban_diagnostics.py`）：通用 distress signal detection——长跑无 heartbeat / 反复 crash / 超时 / spawn 失败 / reclaim。

**CLI**：`hermes kanban {init,task,boards,daemon,...}` ——15 个 verb，见 `hermes_cli/kanban.py:189` 的 subparser 定义。**`/kanban …`** 在 CLI 和 gateway 都接同一套 argparse 树（`kanban.py:2181`）。

详见 [[kanban-multi-agent]]。

### `/goal` — Ralph Loop 一等公民

`hermes_cli/goals.py`（593 行），核心数据：

```python
@dataclass
class GoalState:
    goal: str
    status: str = "active"   # active | paused | done | cleared
    turns_used: int = 0
    max_turns: int = DEFAULT_MAX_TURNS
    last_verdict: Optional[str] = None  # done | continue | skipped
    paused_reason: Optional[str] = None
    consecutive_parse_failures: int = 0
```

**判定模型**（`JUDGE_SYSTEM_PROMPT`，verbatim 摘录）：

> A goal is DONE only when:
> - The response explicitly confirms the goal was completed, OR
> - The response clearly shows the final deliverable was produced, OR
> - The response explains the goal is unachievable / blocked / needs user input (treat this as DONE with reason describing the block).

辅助模型每轮判 done/continue，pause 状态记 `paused_reason`，连续 parse 失败计数防止 judge 故障导致死循环。

持久化在 SessionDB `state_meta`，key = `goal:<session_id>`（`goals.py:175`），所以 `/resume` 切回老 session 时 goal 状态跟着回来。

详见 [[goal-and-ralph-loop]]。
