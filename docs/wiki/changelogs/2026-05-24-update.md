## 1. 安全 Wave — 17 个 commit 一次性补齐 Webhook/Dashboard/平台审批面

时间线异常密集：commit `9732559`–`f378f00` 全部 `2026-05-24 04:24–04:54 -0700`，多数挂 PR `#30737`–`#30746`。统一目标：**关闭 v0.14.0 残余 auth-bypass / 信息泄露 / 不安全默认值**。

### 1.1 Webhook 路由 fail-closed + Svix 签名（PR #30200 + #29629）

`gateway/platforms/webhook.py`：

- **fail closed when secret missing**（`dbf73e9`）：路由 handler 在所有连接前/连接后路径都校验 effective secret，缺失时直接 403 拒绝（`webhook.py:383-395 _validate_signature` 调用前）；之前的 connect-time validator 是软告警，未阻断 hot-reloaded 动态路由。
- **403 不是 500**（`15aa688`）：missing-secret 拒绝路径改返 `403 Forbidden`（`webhook.py:394`），不再触发运维 incident alerting（HTTP 500 通常路由到 PagerDuty）。
- **Svix 签名验证**（`bbf02c3`）：新增 `_validate_svix_signature()`（`webhook.py:690`+），支持 AgentMail / Loops / Resend / Knock 等 Svix-broadcast 类 webhook（header `svix-id` + `svix-timestamp` + `svix-signature`，base64-encoded HMAC，secret 前缀 `whsec_`）。`_header()` 三种大小写都查，`delivery_id` 优先使用 `svix-id`（`webhook.py:489-493`）。
- **限制默认 webhook toolset**（`e4a1220`，PR #30745）：`toolsets.py:75-82` 新增 `_HERMES_WEBHOOK_SAFE_TOOLS = ["web_search", "web_extract", "vision_analyze", "clarify"]`，`"hermes-webhook"` toolset 从 `_HERMES_CORE_TOOLS` 替换为该白名单（`toolsets.py:536`）—— webhook payload 可能含 untrusted 第三方内容（公开 PR 标题/评论），默认不暴露 shell/file/code 执行能力。

### 1.2 Dashboard / API Server 强制 loopback / auth

- **Dashboard WebSocket 强制 loopback**（`9732559`，PR #30741）：删除 `hermes_cli/web_server.py:_is_public_bind()`，`_ws_client_is_allowed()` 不再为 `--insecure` 模式（`bound_host ∈ {0.0.0.0, ::}`）放宽 WebSocket 客户端 IP。`--insecure` 仅用于 HTTP API（仍由 session token 守护），WebSocket 强制只接受 `127.0.0.1 / ::1 / localhost / testclient`（`web_server.py:3296-3305`）。
- **Docker dashboard 默认 loopback**（`2df2f91`，PR #30740）：`docker/entrypoint.sh:111-130` `HERMES_DASHBOARD_HOST` 默认 `127.0.0.1`（之前 `0.0.0.0`），不再自动追加 `--insecure`。要外暴露必须显式覆盖 + 用户自带反向代理。
- **Dashboard 插件 rescan 需 auth**（`ee002e7`，PR #27340）：移除 `hermes_cli/web_server.py` 的 `rescan` route 例外，与其他 dashboard write endpoint 同走 session-token gate。
- **API server placeholder secret 扩展**（`be27bfe`，PR #30738）：`hermes_cli/auth.py:553-560 _PLACEHOLDER_SECRET_VALUES` 新增 `"your_api_key_here"`，常见的环境变量样板值不再被当作有效凭证启用 API server。

## 3. Plugin `register_auxiliary_task()` 新 API（feat，`e752c94`）

`hermes_cli/plugins.py:703-784 PluginContext.register_auxiliary_task(key, *, display_name, description, defaults=None)`。

### 解决什么

Auxiliary LLM tasks（vision / compression / web_extract / approval / mcp / title_generation / skills_hub / curator）原由两处硬编码列表绑定：

- `hermes_cli/main.py:_AUX_TASKS`（picker 显示）
- `gateway/run.py:780-820`（config→env 桥 `AUXILIARY_<KEY>_*`）

Memory / context 类插件想加自己的 auxiliary task slot（如 `memory_retain_filter` 做 hindsight pre-retain dedup）就**必须改核心**。这违反"插件不动核心"约束。

### 解决方式

`PluginContext.register_auxiliary_task(key, ...)`（`plugins.py:703-784`）：

1. 校验 `key` 形状（snake_case，禁止 shadow 内置 8 个 key —— 列表来自 `_BUILTIN_AUX_TASKS`，lazy import 解循环）。
2. 注册到 `self._manager._aux_tasks[key]`，记录 `display_name / description / defaults / plugin`。
3. `gateway/run.py:791-792` 启动时 `get_plugin_auxiliary_tasks()` 汇总后，把每个 plugin 注册的 key 与 builtin 一起循环 bridge config → env（`AUXILIARY_<KEY_UPPER>_{PROVIDER,MODEL,BASE_URL,API_KEY}`）。
4. `hermes model → Configure auxiliary models` picker 自动收录（`main.py:2514` docstring）。

未来 plugin 加 auxiliary task 不再需要触碰 `_AUX_TASKS` 或核心 env-bridging 字典。

## 5. Streaming 完成可见性三连

### 5.1 Guardrail halt 消息抵达 stream client（`38b8d0d` + `186bf25 test`，#30770）

`agent/conversation_loop.py:3478-3486`：tool guardrail 触发（`max_tool_failures` 等）退出 turn 时，原代码只 append final_response 到 messages，**没**通过 stream callback 推给客户端。SSE 流静默关闭，Open WebUI 等客户端看到空 finish chunk —— 与 crash 无法区分。

修复：halt 分支在 `messages.append` 后调用 `agent._safe_print(f"\n{final_response}\n")` 与 `agent.stream_delta_callback(final_response) / stream_delta_callback(None)`，告诉客户端"是 guardrail 主动 halt，不是崩溃"。+49 行新测试（`tests/run_agent/test_tool_call_guardrail_runtime.py`）。

### 5.2 Plugin `transform_llm_output` 流式可见性（5 commits 链）

`transform_llm_output` plugin hook 在 streaming 已经发完后修改 final_response，gateway/ACP 之前都因 `streamed_message=True` 跳过 final send，hook 修改不可见。

修复链：

1. `agent/conversation_loop.py:4081,4100,4152`：新增 `_response_transformed` 标志，hook 返回非空字符串时置 True，写入 result dict。
2. `gateway/run.py:17044` `run_sync` 返回字典 cherry-pick 加入 `response_transformed`（`a4ceead`）。
3. `gateway/run.py:17685-17717`（`5cb21e3`）：`_transformed=True` 时不抑制 final send，但**编辑** streamed message in-place（`stream_consumer.message_id` + `adapter.edit_message(...)`），不发重复消息。
4. `gateway/stream_consumer.py:196-199`：新增 public property `message_id`。
5. `acp_adapter/server.py:1537-1544`（`7eb6c7f` + `60d20a3`）：ACP `final_response and conn and (not streamed_message or result.get("response_transformed"))` —— 只在被 transform 的路径上抹掉 `streamed_message` 屏蔽。

### 5.3 Partial-stream finish_reason 修复（`9140be7` + `20b3703` + `6cafcf9 test`，#30963）

网络中断的 partial stream stub 之前 finish_reason 是默认 `stop`，触发了"正常结束"路径而非长度续传路径。

- `agent/chat_completion_helpers.py:97-141 + 1312-1345`（partial stub 生成处）：text-only partial stream 强制 `finish_reason="length"`，触发 `agent/conversation_loop.py` 的 `length_continue_retries=3` 续传逻辑。
- `conversation_loop.py:42-50` partial-stream 路径专门 prompt 模板（区分网络中断 vs 模型主动 length-truncation），但仍走同样 truncated_response_parts merging。+258 行新测试 pin contract。

## 7. Skills AST 深度诊断（feat，`7255050` + `4254f7d refactor`）

### CLI 接入

### 检测覆盖

动态 import / 属性访问的混淆模式：`importlib.import_module(computed)`, `__import__(computed)`, `getattr(obj, computed)`, `obj.__dict__[computed]`. 这些是 skills_guard regex 难匹配的"运行时构造名"绕过路径。

### 设计选择（refactor `4254f7d`）

原 PR 700+ 行裁到 133 行：

- 三个 entry point（file / dir / bundle）合一 → 单 `ast_scan_path(path)`。
- 删 `AstFinding` dataclass + severity field，改 `(file, line, pattern_id, description)` 元组。
- 输出 label "diagnostic hints"，不是 "security verdict" —— skills_guard 是 security gate，AST 是 review aid，两者职责分离。
