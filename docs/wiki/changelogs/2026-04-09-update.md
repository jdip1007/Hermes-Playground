## New Features

### Structured API Error Classification + Intelligent Failover

**commit**: `1a3ae6ac` (#6514) | **New File**: `agent/error_classifier.py` (789 lines)

Replaced the scattered string-matching logic in `run_agent.py` with a unified error classifier. All API errors are now classified into 13 types of `FailoverReason`, each corresponding to a different recovery strategy:

| Error Type | Recovery Strategy |
|---------|---------|
| `auth` | Refresh/rotate credentials |
| `billing` | Immediately switch Provider |
| `rate_limit` | Backoff wait then rotate |
| `context_overflow` | Compress context |
| `timeout` | Rebuild client + retry |
| `model_not_found` | Fallback to other models |
| `server_error` | Retry |

The classification result includes structured recovery hints (`retryable`, `should_compress`, `should_rotate_credential`, `should_fallback`). The retry loop directly reads these fields, eliminating redundant error parsing.

### Unified Execution Layer (spawn-per-call)

**commit**: `d684d7ee`

Terminal environment architecture refactoring. All 6 backends (local, docker, ssh, singularity, modal, daytona) are unified into a "independently spawn `bash -c` process per command" model:

- Capture a session snapshot once during initialization (env vars, functions, aliases)
- Re-source the snapshot before each command
- CWD is persisted across calls via stdout markers (remote) or temporary files (local)

This replaces the previous persistent shell implementations maintained separately by each backend (`persistent_shell.py` has been deleted).

---

### Tool Result Persistence (3-Layer Protection)

**New Files**: `tools/tool_result_storage.py`, `tools/budget_config.py`

A three-tier mechanism to prevent large tool outputs from blowing up the context window:

| Tier | Location | Mechanism |
|------|------|------|
| 1. In-tool truncation | Inside each tool | Tools pre-truncate their own output |
| 2. Single result persistence | `tool_result_storage.py` | Exceeds threshold (default 100K chars) → writes to `/tmp/hermes-results/`, context only retains a 1500-char preview + file path |
| 3. Turn-level aggregate budget | `budget_config.py` | Total tool results in a single turn exceed 200K chars → largest non-persisted result overflows to disk |

`read_file` is pinned with threshold `∞` to prevent an infinite loop of "persist → read → persist again".

## Important Fixes

### Stability

| Fix | commit | Impact |
|------|--------|------|
| Session deletion changed to orphaning | `a9409990` (#6513) | When deleting/cleaning sessions, child sessions (created from compression splits) are no longer cascade-deleted. Instead, they become orphans (`parent_session_id` set to NULL), preserving searchable history |
| Empty response retries 3 times | `1eabbe90` (#6488) | Automatically retries when the model returns empty content, instead of failing immediately |
| Credential exhaustion TTL reduced to 1 hour | `b408379e` (#6504) | Reduced from 24 hours to 1 hour, allowing faster recovery attempts after a key is marked as invalid |
| Compression tail protection min_tail=3 | `d12f8db0` | Forces retention of at least 3 tail messages during compression, based on token budget rather than fixed message count |
| Improved compression quality | `85670314` | Named constants, tool call tracking, degradation alerts; retains more critical information after compression |
| turn-exit diagnostic logs | `268ee6bd` (#6549) | Logs the reason for every agent loop exit, facilitating debugging |
| flush_memories respects config timeout | `42e366f2` | No longer hardcodes 30 seconds, now reads from config |

## Architecture Changes Summary

1. **Error Handling**: From 190 lines of scattered string matching in `run_agent.py` → centralized structured classifier in `error_classifier.py`
2. **Terminal Execution**: From each backend maintaining its own persistent shell → unified spawn-per-call + session snapshot
3. **Tool Results**: Added 3-tier overflow protection, large outputs no longer blow up the context window
4. **Session Management**: Deletion operation changed from cascade delete to orphaning, protecting historical data integrity
