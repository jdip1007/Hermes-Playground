## 6. write_file / patch 原子化 + UTF-8 BOM

### 6.1 原子写（#35252，`39f6b6e9d`）

`tools/file_operations.py:772 _atomic_write(self, path, content)` + `:1193-1207` patch 路径：

- 流到**同目录** temp file，再 `mv` 到 target（同 FS atomic rename）。
- 避免跨 device 非原子 copy；保留正确 mode bits（不静默扩/收权限）。
- 部分写入崩溃后 target 仍是旧版本，不会留半成品。

### 6.2 UTF-8 BOM 处理（#35278，`5f84c9144`）

某些 Windows 编辑器在文本文件首字符塞 U+FEFF。原行为：read_file 把 BOM 当成首字符返回；patch 匹配真正首行可能漏；write/patch round-trip 静默剥 BOM。

**新行为**（`tools/file_operations.py`）：

- `:127 _UTF8_BOM = "﻿"` + `:131 _strip_leading_bom(text) -> (text, had_bom)`（只剥首 BOM，中段 U+FEFF 保留）。
- `:142 _starts_with_utf8_bom(text)` 探测。
- `:848` 文件是否带 BOM 的属性，read 时 `:947` "Strip a leading UTF-8 BOM so the model never sees a phantom U+FEFF"。
- write/patch 探测原文件 BOM 状态，round-trip 保留。

## 11. Streaming cumulative-resend 修复 → revert

**双 commit 拉锯**：

- **`ca03486b6 fix(streaming): stop duplicating tool-call args from cumulative-resend providers (#35718)`**：DeepSeek / Baidu Qianfan 在 stream tool-call 参数时是**累积模式**——每个 chunk 重发"截至目前完整 args"，而非新片段。累加器盲目 `+=` 拼成 `'{...}{...}{...}'`，json.loads 失败被清成 `{}`。修复加每槽 cumulative-resend latch：delta 是 prev 严格超集（`len(_new) > len(_prev)` 且 `_new.startswith(_prev)`）就触发。
- **`2b5268f71 revert: drop cumulative-resend tool-arg heuristic from shared streaming path (#35718) (#35860)`**：6 小时后 revert——latch 误触正常 provider 的 delta 模式（部分 delta 也会偶然满足 startswith），把正常增量误判为累积重发，反而漏数据。

**最终状态**：共享 streaming path 不带 cumulative-resend 启发式；DeepSeek/Qianfan 的修复需另开针对 provider 的局部分支。
