# Hermes Agent 更新日志（2026-04-30 ~ 2026-05-06）

跨越 **v0.12.0 (v2026.4.30)** 大版本发布 + 一周后续补丁，共 **211 commits**（自 2026-04-29 以来）。

核心主题：**Provider 系统插件化**（架构级，33 providers 全迁移）+ **Kanban 多 Agent 协作板**（新模块）+ **Checkpoints v2** 单仓重写 + **Web Tools 按能力拆分** + **Browser Lightpanda 引擎** + **i18n 系统** + 一系列稳定性修复。

> **注**：v0.12.0 大版本的主要内容（Curator、Self-improvement loop、Spotify、Google Meet、Microsoft Teams、ComfyUI/TouchDesigner bundled、TUI LaTeX、Vercel Sandbox 等）已在 [2026 04 29 Update](2026-04-29-update.md) 中详述。本页聚焦此次发布之后落地的**新架构**与**新模块**。

## 架构级变更

### Provider 系统全面插件化（重大）

`providers/__init__.py` + `providers/base.py` 引入 **`ProviderProfile` ABC**，把原先散落在 `agent/auxiliary_client.py`、`agent/run_agent.py`、`agent/model_metadata.py` 等处的 provider 行为（auth、endpoint、headers、temperature、reasoning 翻译、fallback model）**集中到声明式 dataclass**。每个 provider 一个 `__init__.py` 调用 `register_provider(profile)` 即注册。

**33 个 profiles 在 28 个 plugin 目录下**（gemini/kimi-coding/opencode-zen 各 2 个、minimax 3 个；其余 24 个一目录一 profile）：

```
plugins/model-providers/
├── ai-gateway/      ├── alibaba/             ├── alibaba-coding-plan/
├── anthropic/       ├── arcee/               ├── azure-foundry/
├── bedrock/         ├── copilot/             ├── copilot-acp/
├── custom/          ├── deepseek/            ├── gemini/   (2 profiles)
├── gmi/             ├── huggingface/         ├── kilocode/
├── kimi-coding/ (2) ├── minimax/ (3)         ├── nous/
├── nvidia/          ├── ollama-cloud/        ├── openai-codex/
├── opencode-zen/(2) ├── openrouter/          ├── qwen-oauth/
├── stepfun/         ├── xai/                 ├── xiaomi/    └── zai/
```

**发现链**（`_discover_providers()`）：
1. `<repo>/plugins/model-providers/<name>/` — bundled
2. `$HERMES_HOME/plugins/model-providers/<name>/` — user (last-writer-wins，可覆盖 bundled)
3. `providers/<name>.py` — legacy 单文件兼容路径

**用户插件优先级最高**——任何人都可以 monkey-patch 或替换内置 profile，不必改 repo 代码。

详见 [Provider Transport Architecture](../concepts/provider-transport-architecture.md)、[Smart Model Routing](../concepts/smart-model-routing.md)。

## 新模块：Kanban 多 Agent 协作板

`hermes_cli/kanban.py`（2036 行）+ `hermes_cli/kanban_db.py`（4087 行）+ `hermes_cli/kanban_diagnostics.py`（649 行）+ `tools/kanban_tools.py`（855 行）+ `plugins/kanban/dashboard/` —— 跨 profile / 跨 project 的多 agent 任务编排。

### 设计

- **共享根**：默认 DB 在 **shared Hermes root**（profile 父目录）`<root>/kanban.db`，让多个 profile 落到同一块板上协调。
- **多 board (project) 支持**：可在 `<root>/kanban/boards/<slug>/` 创建独立 board，各自有 `kanban.db` / `workspaces/` / `logs/`。Worker 被 dispatcher pin 到 task 所在 board，看不到其他 board。
- **第一块 board 仍叫 `default`**：DB 为 `<root>/kanban.db`（不是 `boards/default/kanban.db`），向后兼容。
- **Board 解析顺序**：`board=` 参数 > `HERMES_KANBAN_BOARD` > `HERMES_KANBAN_DB` > `<root>/kanban/current` > `default`。

### 7 个 Worker Tools（仅在 worker 上下文注入）

`tools/kanban_tools.py:_check_kanban_mode()` 双门控：
1. `HERMES_KANBAN_TASK` 环境变量（dispatcher 启动 worker 时注入），或
2. 当前 profile 的 toolsets 包含 `"kanban"`（orchestrator profile，如 techlead）

满足任一即注册 7 个工具：
| Tool | 用途 |
|------|------|
| `kanban_show` | 读取当前 task 详情 |
| `kanban_complete` | 标记完成 |
| `kanban_block` | 标记阻塞 |
| `kanban_heartbeat` | 长跑 worker liveness 信号 |
| `kanban_comment` | 写评论 |
| `kanban_create` | 子卡（worker-created card）|
| `kanban_link` | 跨 task 链接 |

**为什么是 tool 而不是 shell**：worker 终端可能指向 Docker / Modal / Singularity / SSH，里面没有 `hermes` 也访问不到 `~/.hermes/kanban.db`。Tool 在 agent 进程里跑，无论后端都能落库。

### CLI 15 动词

`hermes kanban {init, boards, create, list, show, assign, reclaim/reassign, diagnostics, link, unlink, claim, comment, complete, block, unblock, archive, tail, dispatch, daemon, watch, stats, notify, log, runs, heartbeat, assignees, context, gc}`

- `dispatch_in_gateway: true`（默认）—— Gateway 内嵌 dispatcher，systemd `hermes-kanban-dispatcher.service` 标记 **DEPRECATED** 但保留给无法跑长 service 的环境。
- 限并发：`gateway: kanban.max_spawn`。
- Hallucination gate：worker 声称创建的 card 若不存在，dispatcher 触发恢复 UX。

### 诊断引擎（distress signals）

`hermes_cli/kanban_diagnostics.py` 引入 `Diagnostic` dataclass：每个 diagnostic 携带 **kind**（canonical code）、**severity**（warning/error/critical）、**title/detail**、**suggested actions**（dashboard 渲染成按钮，CLI 渲染成提示）。规则**无状态、只读**——按需在 `/board` 加载、`/tasks/:id` fetch、`hermes kanban diagnostics` 触发。**自动消失**：底层失败模式恢复后停止触发，audit event trail 留档。

### Dashboard 插件

`plugins/kanban/dashboard/`：tab `/kanban`（在 `after:skills`），FastAPI router 挂在 `/api/plugins/kanban/`，WebSocket 推 `task_events`，要求 `?token=` 查询参数（`/api/plugins/` 跳过 dashboard auth middleware，但 WebSocket 仍需 session token）。

## Checkpoints v2 — 单仓重写

`tools/checkpoint_manager.py`（1638 行）从"每个工作目录一个 shadow git 仓库"改为 **`~/.hermes/checkpoints/store/` 单一共享 bare 仓库**。Object DB 跨所有工作目录去重——同 repo 的 12 个 worktree 几乎不增加磁盘占用。

```
~/.hermes/checkpoints/
├── store/                          # 单一共享 git 仓库
│   ├── HEAD, config, objects/      # 标准 git 内部
│   ├── refs/hermes/<hash16>        # per-project branch tip
│   ├── indexes/<hash16>            # per-project git index
│   ├── projects/<hash16>.json      # {workdir, created_at, last_touch}
│   └── info/exclude
├── .last_prune                     # 自动 prune 幂等标记
└── legacy-<timestamp>/             # 自动迁移的 v1 旧 shadow repos
```

**修复 v1 的三大问题**：
1. **零跨项目去重** → 单仓 git object DB 自然去重
2. **`_prune()` 是 no-op**（`max_snapshots` 仅限制 `/rollback` 列表，loose objects 永远累积）→ 真 prune
3. **默认 `enabled=True, auto_prune=False`** → 用户没用 `/rollback` 也付出磁盘成本

实际现场报告：单工作站 47 个 shadow repos 占 847 MB，多为 hermes-agent 源码 tree 的冗余克隆。

`prune_checkpoints` 删除 orphan refs（工作目录不存在）+ stale refs（`retention_days` 过期），跑 `git gc --prune=now`，再按 `max_total_size_mb` size-cap 删最老 checkpoint。

## Web Tools — 按能力拆分

`tools/web_providers/` 引入 **`WebSearchProvider` / `WebExtractProvider` ABC**，把 search 和 extract 拆开按能力选 backend：

```yaml
web:
  backend: "firecrawl"       # shared fallback
  search_backend: "searxng"  # search 用 SearXNG
  extract_backend: "firecrawl"  # extract 用 Firecrawl
```

**优先级**（每能力独立）：
1. `web.search_backend` / `web.extract_backend`（显式 per-capability）
2. `web.backend`（共享 fallback）
3. 自动检测 env vars

### 新增 SearXNG search-only backend

`tools/web_providers/searxng.py`：自托管隐私搜索引擎，**无 API key**，只需 `SEARXNG_URL`（如 `http://localhost:8080`）。
- 注册为 backend 候选（优先级低于付费 service）
- `web_extract` / `web_crawl` 命中 SearXNG 时返回明确错误（"search-only backend"）
- `hermes tools` picker 列出，`hermes doctor` 检查可用性
- 23 个测试覆盖 happy path / score sorting / limit / HTTP errors

## File Tools — Post-Write Delta Lint

`tools/file_operations.py` 添加 **in-process linters** for `.py` / `.json` / `.yaml`/`.yml` / `.toml`（`LINTERS_INPROC` dict）：

```python
LINTERS_INPROC = {
    '.py': _lint_python_inproc,    # ast.parse
    '.json': _lint_json_inproc,    # stdlib json
    '.yaml': _lint_yaml_inproc,
    '.yml': _lint_yaml_inproc,
    '.toml': _lint_toml_inproc,
}
```

**post-first / pre-lazy 模式**（借鉴 Cline + OpenCode）：
1. 先 lint 写入后状态
2. 仅当 post 报错且 pre_content 已抓取时，才 lint pre 状态做 diff
3. 若 pre 已有相同错误 → 报告 "still broken, pre-existing"（`success=False` 但解释错误是预存的）
4. 若引入了真新错误 → 暴露新错误，过滤掉预存的

`write_file()` 对 in-process-lintable 扩展名捕获 pre_content；shell linter（`py_compile` / `node --check` / `tsc` / `go vet` / `rustfmt`）保留为无 in-process 等价物的语言的 fallback；`patch` 同样接入。

## Browser — Lightpanda 引擎

`tools/browser_tool.py` 新增 `--engine lightpanda` 支持（agent-browser v0.25.3+）：

```yaml

  engine: "lightpanda"  # auto | lightpanda | chrome
```

或 `AGENT_BROWSER_ENGINE` 环境变量。`auto`（默认）= 不传 `--engine`，agent-browser 默认 Chrome。

## i18n 系统（新）

`agent/i18n.py`（233 行）+ `locales/{en,zh,ja,de,es,fr,tr,uk}.yaml` —— **轻量级静态消息翻译**，仅覆盖最高影响的用户面字符串：

- approval prompts
- gateway slash command 回复

**不翻译**：agent 输出、log、traceback、tool 输出、slash-command description。

**语言解析顺序**：
1. 显式 `lang=` 参数
2. `HERMES_LANGUAGE` 环境变量
3. `display.language` from `config.yaml`

支持 8 种语言：**en/zh/ja/de/es/fr/tr/uk**。包含别名映射（`chinese`/`zh-CN`/`mandarin` → zh，`日本語`/`jp` → ja，等等）。

未知语言静默回落到 en。Catalog 缺失 key 回落 en，再缺则返回 key path 自身（不会因为 catalog 损坏崩溃）。

## Skill 系统：Pin 语义收窄

`fix(skills): pin protects against deletion only, not edits` (#20220)：

之前 `_pinned_guard()` 是 hard fence，**任何 skill_manage 写操作（edit/patch/delete/write_file/remove_file）都拒绝**。

设计混淆了两件事：
1. **Pin as deletion protection**——保护 skill 不被 curator/agent 删除（用户实际想要的）
2. **Pin as content freeze**——禁止 agent 改 skill（造成 unpin → patch → re-pin 麻烦舞步，鼓励 skill 失修）

**新行为**：`_pinned_guard()` 仅在 `skill_manage(action='delete')` 路径触发。Patch / edit / supporting-file write **全部放行**。Curator 自身的 pinned-skip 行为不变（`agent/curator.py:271` 自动归档 + `:349` LLM review prompt 都仍跳过 pinned）。

## Memory：Hindsight Provider（新）

`plugins/memory/hindsight/__init__.py` —— **长期记忆** with knowledge graph、entity resolution、多策略检索。MemoryProvider ABC 实现，原 PR #1811 by benfrank241 改造。

**两种模式**：
- **Cloud**：`HINDSIGHT_API_KEY` + `HINDSIGHT_API_URL`
- **Local**：嵌入 daemon，`HINDSIGHT_IDLE_TIMEOUT` 控制空闲关闭（默认 300s，0 = 不关）

**配置 env vars**（精选）：
| Env | 默认 | 说明 |
|------|------|------|
| `HINDSIGHT_API_KEY` | — | Cloud API key |
| `HINDSIGHT_BANK_ID` | `hermes` | memory bank id |
| `HINDSIGHT_BUDGET` | `mid` | recall budget: low/mid/high |
| `HINDSIGHT_MODE` | `cloud` | cloud / local |
| `HINDSIGHT_TIMEOUT` | 120s | API 请求超时 |
| `HINDSIGHT_RETAIN_TAGS` | — | 逗号分隔标签 |

`feat(hindsight): probe API for update_mode='append' support` —— 启动时 probe `<api_url>/version`，gate 在 Hindsight ≥ 0.5.0。
- **支持时**：用稳定 session-scoped `document_id = session_id` + `update_mode='append'`，跨进程对同一 session 的 retain 合并到一条文档（不再产生 N 条 process-stamped 重复）

## API Server — X-Hermes-Session-Key

`gateway/platforms/api_server.py:_parse_session_key_header()` —— OpenAI 兼容端点（Open WebUI、自定义 web UI）现在可以传 **`X-Hermes-Session-Key`** header，提供**稳定 per-channel 标识**，scope 长期记忆（Honcho、Hindsight 等）独立于 transcript-scoped `X-Hermes-Session-Id`。

匹配原生 gateway 的 `session_key` / `session_id` 拆分语义：**一个 channel 一个稳定 key，多个 transcript 在 `/new` 时旋转**。

- 三个端点都 honor：`/v1/chat/completions`、`/v1/responses`、`/v1/runs`
- JSON / SSE 响应回声 header
- `/v1/capabilities` 通告 `session_key_header` 让 client feature-detect
- 同样的 API-key gate / 控制字符 sanitization / 256 字符长度上限

## Auth — Profile→Global Fallback

`fix(auth): fall back to global-root auth.json for providers missing in profile` —— Profile 进程（kanban worker、cron 子进程、delegated subagent）只读 profile 的 `auth.json`。若某 provider 在全局认证但 profile 内没有，credential_pool 返回空，进程报 `'No LLM provider configured'`。#18594 修了 `HERMES_HOME` 传递，反而暴露了这个问题——worker 现在落到正确 profile 里，profile 影子全局且无 fallback。

**新语义**（read-only，per-provider shadowing）：
- Profile 有 provider X 任何条目 → 只用 profile（忽略 global）
- Profile 无条目 → fall back 到 global
- 写入（`write_credential_pool` / `_save_auth_store`）仍写 profile
- Classic mode（`HERMES_HOME == global root`）跳过 fallback（`_global_auth_file_path()` 返 None）

`get_provider_auth_state` 同步 mirror，让 OAuth singleton（nous / minimax-oauth / openai-codex / spotify）干净继承。Nous shared-token store (#19712) 仍是 Nous 的权威路径。

## Gateway — Per-Platform 重启通知

`PlatformConfig.gateway_restart_notification`（默认 True）：

- 控制 **重启 ping**（`♻ Gateway restarted`，发到 `/restart` 来源 chat）
- 控制 **启动 ping**（`♻️ Gateway online`，发到 home channel）
- **同时控制**预重启 drain 通知（`⚠️ Gateway restarting — Your current task will be interrupted...`）

设计动机：**operator vs end-user surfaces**。Telegram 这种 back-channel 保留 ping 合理；与 end-user 共享的 Slack workspace 把 "Gateway restarting" 读成 "the bot is broken"，operator 应能一致禁用三种 lifecycle 噪音。

## Reasoning 抽取修复

`fix: prevent stale reasoning from being reused across turns` ([efe1cb0](#)) —— `run_conversation()` 的 reasoning-box 抽取循环之前**反向遍历整个消息历史**找任何带非空 `reasoning` 字段的 assistant 消息。当前 turn 没产生 reasoning（如 provider 返回 `reasoning_content=null`）时，循环走过当前 turn，**显示数分钟/数小时前的 reasoning**。

**修复**：遍历到当前 turn 起始的 user 消息就停。结果：
- Tool-calling turn 中 reasoning 落在 tool-call 步骤、final-answer 步骤 reasoning=None（Claude thinking / DeepSeek v4 / Codex Responses 常见）→ 拿到 turn 内最近的 reasoning
- 当前 turn 真无 reasoning → 干净返回 None
